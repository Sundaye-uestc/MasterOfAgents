开发阶段总览

Phase 0   平台接入验证

目标：初始化工程，验证 Claude Code 可编程接入方式，选择并验证第二平台，跑通统一 AgentEvent，确认进程启停/超时/错误处理/并发控制。

完成标志：至少一个真实 Agent 平台能稳定接入；Claude Code 明确采用 SDK 或 CLI 子进程方案；后续功能可围绕统一 Agent Adapter 开发。

Phase 1   Web 单聊闭环

目标：搭建 Web + Server + Shared 基础工程；完成数据库、REST API、WebSocket 和基础状态管理；实现 1v1 单聊流式回复；支持会话搜索/归档、消息 Pin、引用回复、重新生成；记录 workspace 快照和文件变更。

完成标志：Web 端可完成稳定单聊体验；消息和运行状态可持久化，刷新后可恢复；可展示基础文件变更或 Diff 产物。

Phase 2   群聊协作

目标：实现 Planner LLM 生成结构化 TaskPlan；实现 Orchestrator 任务调度、失败降级和结果聚合；支持群聊成员管理和 @ Agent；展示 Agent 状态、能力标签、run/task/tool 状态；支持对话式创建 Custom Agent。

完成标志：群聊中可分派至少两个 Agent/任务；Planner 失败时能降级为单 Agent 执行；前端可实时展示多 Agent 协作状态。

Phase 3   产物预览与部署

目标：完善 ArtifactService 和 WorkspaceService；支持 Diff 查看/应用/回滚；支持网页 iframe 预览；支持 workspace 打包下载；支持本地静态预览或基础部署状态卡片；补齐权限审批和审计日志。

完成标志：用户可查看、应用、回滚 Agent 生成的产物；可获得下载包或本地预览 URL；产物链路可支撑 Demo 展示。

Phase 4   体验完善与稳定性

目标：修复短期记忆（buildAgentContext + 4000 字符窗口）、Diff 展示完善（unified diff + DiffBlock 渲染器）、WorkspacePanel 集成（文件树 API + 快照回滚 + 面板拖拽）、富媒体内联展示、稳定性修复（EPIPE 崩溃、Agent 卡死看门狗、FileChange 去重）。

完成标志：Web 端核心路径可稳定演示；EPIPE/卡死等问题已修复；富媒体卡片正常渲染。

Phase 5   PPT 生成能力集成

目标：NanoBanana PPT 脚本迁移到 AgentHub/ppt/；实现双路径 PPT 生成（pptxgenjs 秒级 + Gemini AI 图示）；PPTX → HTML 预览转换；前端 PptxViewerCard（Canvas 渲染）+ ImageSlideshowCard（PNG 轮播）；Agent 自审查行为约束。

完成标志：用户可通过聊天请求 Agent 生成 PPT 并预览；双路径均正常工作。

Phase 6   Agent 管理与 Planner 修复

目标：Agent 管理 CRUD（对话式创建 + System Prompt 编辑/AI 润色 + 10 工具集自动匹配）；Planner 修复（ESM hoisting API Key、Prompt 强化、DAG 调度验证、上下文隔离、闲聊识别）；Monaco Editor 集成（多标签页 + Ctrl+S 保存 + 语法高亮）；群聊 UX 优化（GroupAvatar 拼图头像 + 成员管理 ··· 菜单）。

完成标志：用户可创建/编辑/删除自定义 Agent；Planner 分工准确且全中文输出；代码编辑器可正常编辑保存。

Agent Adapter 设计

export interface AgentPlatformAdapter {
  id: string;
  name: string;
  kind: "claude-code" | "codex" | "opencode" | "custom";

  /** 准备环境（安装依赖、检查 CLI 等） */
  prepare(config: AgentInstanceConfig): Promise<void>;

  /** 执行一次 run，产出归一化的 AgentEvent 流 */
  run(input: AgentRunInput): AsyncIterable<AgentEvent>;

  /** 停止一个活跃的 run */
  stop(runId: string): Promise<void>;

  /** 清理适配器资源 */
  dispose(): Promise<void>;
}

AgentRunInput

export interface AgentRunInput {
  runId: string;

  conversationId: string;
  agentId: string;
  prompt: string;
  history: ChatMessage[];
  workspace: WorkspaceRef;

统一 AgentEvent（10 种事件）

export type AgentEvent =
  | { type: "run_started";        runId, agentId }
  | { type: "text_delta";         runId, messageId, content }
  | { type: "thinking";           runId, content }
  | { type: "tool_call";          runId, toolCallId, name, input }
  | { type: "tool_result";        runId, toolCallId, status, output?, error? }
  | { type: "permission_required"; runId, request }
  | { type: "file_change";        runId, change }
  | { type: "artifact_created";   runId, artifactId }
  | { type: "run_completed";      runId, usage? }
  | { type: "run_failed";         runId, code, message };

关键决策：CLI 子进程 vs SDK

Phase 0 验证结论：Claude Code SDK 不稳定，采用 CLI 子进程方案。通过 ProcessSupervisor 管理子进程生命周期，stream-json 格式解析输出，归一化为统一 AgentEvent 流。这是项目最关键的架构决策之一。

ProcessSupervisor（实际配置）

export interface ProcessSupervisor {
  start(spec: ProcessSpec): AsyncIterable<ProcessEvent>;
  stop(processId: string): Promise<void>;

运行流与状态机

单聊流程

message:send → persist user message → create run(mode=direct) → create before snapshot → AgentRuntime.start(adapter) → normalize AgentEvent → persist message/tool/file/artifact → push WS events → create after snapshot → complete run

群聊流程

message:send → persist user message → create run(mode=orchestrated) → PlannerService.generateTaskPlan() → invalid plan? fallback to direct single-agent run → persist tasks → schedule ready tasks by DAG → AgentRuntime.start(task.agent) → persist outputs/file changes → aggregate result → post orchestrator summary → complete run

Run 状态机

queued → running → completed
running → waiting_permission → running
running → failed | cancelled

Task 状态机

queued → running → completed
running → blocked | failed | skipped

Orchestrator 调度设计

Planner LLM

默认模型GPT-4o-mini / Claude Haiku / 同级轻量模型
输入用户消息、pinned messages、最近历史、成员 Agent、能力标签、workspace 文件摘要
输出严格 JSON，必须匹配 TaskPlan schema
校验失败重试 1 次；仍失败则降级为单 Agent 直接执行
export interface TaskPlan {
  runId: string;
  summary: string;
  strategy: "single" | "sequential" | "parallel" | "mixed";
  tasks: Array<{
    id: string;
    title: string;
    description: string;
    agentId: string;
    dependencies: string[];
    expectedOutput: string;
    writeScope?: string[];
    riskLevel: "low" | "medium" | "high";
  }>;
  confirmationRequired?: boolean;
}

调度规则

  1. 依赖完成后进入 ready 队列
  2. 写入范围重叠的任务串行
  3. 无写入冲突的任务可并行
  4. 高风险任务进入 waiting_permission
  5. 子任务失败默认重试 1 次，再尝试同能力 Agent

Planner System Prompt 约束

  • 只输出 JSON，不输出 Markdown
  • tasks[].agentId 必须来自候选 Agent 列表
  • dependencies 只能引用同一 plan 内 task id
  • 高风险操作设置 confirmationRequired = true
